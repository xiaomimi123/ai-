# 行政事业单位文档合规协同检查智能体

## 自研技术开发文档 v1.0（脱离 Dify）

> 本文档用于指导自主开发，不依赖 Dify 等第三方 AI 平台。
> 开发流程：本地 Docker 测试 → 部署到服务器 → 交付客户使用。

---

## 一、架构总览

完全自研，不依赖 Dify 等平台。整体分四层：

| 层级 | 技术选型 | 职责 |
|------|---------|------|
| **前端层** | Next.js + React + TailwindCSS | 界面、文件上传、检查任务管理、问题台账、报告导出 |
| **应用服务层** | Python FastAPI | 业务逻辑、任务编排、权限控制（自研相比 Dify 多出来的主要工作量） |
| **AI 能力层** | 自建 RAG + LLM 调用 + 联动校验引擎 | 知识检索、合规分析、跨文件交叉比对 |
| **数据层** | PostgreSQL + Qdrant + 对象存储 | 业务数据/台账、向量知识库、原始文件 |

**为什么自研适合这个项目：** 本需求的核心是「跨文件联动校验引擎」（招标↔投标↔评标↔合同的数据交叉比对），这是高度定制的多文档比对逻辑，Dify 的标准 RAG 工作流难以表达，自己写反而更直接、更可控。

---

## 二、技术选型

| 组件 | 选型 | 说明 |
|------|------|------|
| 后端框架 | FastAPI | 异步、适合 AI 任务、自动生成 API 文档 |
| 向量库 | Qdrant（推荐）或 ChromaDB | Qdrant 性能好、支持 metadata 过滤（分类检索刚需）；Chroma 更轻量 |
| Embedding | bge-large-zh-v1.5（推荐）或 OpenAI text-embedding-3 | bge 中文效果好、可本地部署、免外币 |
| LLM | Claude 主用 | 中文法规理解、长文本、结构化输出稳定；通过适配层封装便于切换 |
| PDF 解析 | PyMuPDF | 普通 PDF |
| OCR | PaddleOCR | 扫描件 PDF / 图片 |
| Word 解析 | python-docx | — |
| Excel 解析 | openpyxl | — |
| 任务队列 | Celery + Redis | 批量检查、联动校验是耗时任务，必须异步 |
| 数据库 | PostgreSQL | 业务数据、问题台账 |
| 对象存储 | MinIO（本地）/ 云 OSS | 原始文件存储 |

## 三、核心模块设计

### 3.1 文档分类与管理模块

按需求固定 **9 大一级分类**：

1. 内部制度
2. 合同
3. 采购招标（含招标 / 投标 / 评标三个子类）
4. 内控报告
5. 决算报告
6. 财务报告
7. 国有资产报告
8. 绩效评价报告
9. 其他佐证资料

每个文档存储时打上 metadata：`category`（一级分类）、`subcategory`、`project_id`、`year`、`department`、`batch`（检查批次）。这套 metadata 既用于前端二次标签检索，也用于向量检索时的过滤条件。

### 3.2 文档解析模块

设计成**统一入口 + 策略分发**：根据文件扩展名路由到对应解析器，统一输出结构化文本（带页码/章节位置信息——后续问题定位需要标注「资料具体位置」）。解析结果缓存，避免重复解析。

解析输出统一结构：

```json
{
  "text": "全文文本",
  "page_blocks": [{"page": 1, "section": "第一章", "content": "..."}],
  "tables": [],
  "metadata": {}
}
```

让位置信息贯穿全流程。

### 3.3 RAG 知识库模块（自研重点）

脱离 Dify 后要自己搭的核心。流程：

```
法规/规则文档 → 分块(chunking) → 向量化(embedding) → 存入 Qdrant(带分类 metadata)
查询时：查询向量化 → 按 category 过滤 + 向量相似度召回 Top-K
```

**分块策略**针对法规文档特别处理：按「条款」切分而非固定长度，每个 chunk 保留法规名称、条款号作为 metadata，检索回来能直接引用「《政府采购法》第 X 条」作为依据。

知识库分两类：
- **法规库**：上位法、财经规范（《政府采购法》《政府会计制度》《行政事业单位内部控制规范》等）
- **规则库**：7 套检查模板、高频风险清单

### 3.4 检查规则引擎

需求区分「刚性规则」和「柔性规则」，引擎对应两条路径：

**刚性规则**（格式要素、流程留痕、签章完整性等）走**确定性校验**——规则配置 + 文本匹配/正则/字段提取判断，不需要 LLM，快且准。例：「发文文号是否存在」「大小写金额是否一致」。

**柔性规则**（条款对标上位法、权责匹配、逻辑合理性）走 **LLM + RAG**——文档内容 + 检索到的法规组装成 prompt，让 LLM 输出疑点。

**预置 7 套模板**（开箱即用，支持自定义增补）：

| 模板 | 适用文件 |
|------|---------|
| 制度合规检查模板 | 内部制度 |
| 合同全流程检查模板 | 合同 |
| 招采三合一检查模板 | 招标/投标/评标（联动） |
| 内控报告检查模板 | 内控报告 |
| 财务+决算联合检查模板 | 财务报告、决算报告 |
| 资产报告检查模板 | 国有资产报告 |
| 绩效评价报告检查模板 | 绩效评价报告 |

支持一键启用、批量套用、按单位制度增补条款。

### 3.5 跨文件联动校验引擎（最核心、最具差异化）

Dify 最难实现、自研最有价值的部分。设计三条联动链：

- **招采链**：招标 → 投标 → 评标 → 合同，比对标的、参数、报价、工期、服务内容一致性
- **财务链**：财务报告 → 决算报告 → 资产报告 → 合同付款，数据交叉互验
- **报告链**：内控报告 → 绩效报告 → 项目资料，内容交叉印证

**实现思路（先抽取、后比对，分两阶段）：**

1. 用结构化抽取（LLM 配合 JSON schema）从每份文档提取关键字段（合同金额、中标价、招标预算等），存入结构化比对表
2. 做字段级比对 + 语义比对，不一致处高亮标注

分两阶段比让 LLM 一次性读完所有文档更准、更省 token。

### 3.6 问题台账与报告模块

按文件类型分别生成台账（制度问题台账、招采问题台账、报告类台账）。

统计报表：完整性统计、高频违规 TOP 排行、跨文件不一致专项清单。

报告导出用 docx-js（前端生成）或 python-docx（后端生成），支持招采专项报告、综合报告、全套总报告三种。

**每个问题条目统一结构：** 疑点描述、资料位置（文件名+页码+段落）、法规依据、问题类别、风险等级、整改建议。

### 3.7 协同与权限模块

角色：管理员、招采专员、财务专员、内控专员。权限按文件分类细分（财务岗只看财务类、采购岗只看招采合同类）。

工作流：

```
任务创建 → 资料归集 → 智能初检 → 人工协同复核(在线批注/交叉复核/逐级流转)
       → 问题下发 → 整改复核销号 → 汇总归档
```

全程留痕可溯源，每个操作记日志。

## 四、关键技术难点与对策

| 难点 | 对策 |
|------|------|
| 跨文件字段抽取准确率 | 每类文档定义明确抽取 schema，用 LLM JSON mode 强约束输出，抽取后规则校验兜底 |
| 长文档超上下文窗口 | 分块处理，按章节/页码分段调 LLM 后合并；或先 RAG 召回相关段落再分析 |
| 扫描件 OCR 精度 | 建议客户优先提供电子版；扫描件配 PaddleOCR 并人工抽检 |
| AI 幻觉/误报 | 定位「辅助工具」，强制人工复核；prompt 约束「只引用检索到的法规，不得编造条款」 |
| 性能 | 刚性规则尽量不走 LLM，确定性能做的别交给模型 |

## 六、开发阶段建议

按以下顺序推进，先稳基础再攻难点：

| 阶段 | 目标 | 说明 |
|------|------|------|
| **第一阶段** | 单文件检查闭环 | 解析→刚性校验→RAG+LLM 柔性校验→问题台账→报告导出。先跑通一类文档（建议「合同」或「内部制度」，规则清晰） |
| **第二阶段** | 分类扩展 | 扩展到 9 大分类和 7 套模板 |
| **第三阶段** | 联动校验引擎 | 先做招采链（最具代表性），再做财务链、报告链 |
| **第四阶段** | 协同与适配 | 协同复核、权限细分、批量处理、迎检适配 |

联动校验引擎最难，放在基础设施稳定后再做。

## 七、本地 Docker 测试流程

前期在本地用 Docker 完成开发和测试。

### 7.1 docker-compose 服务编排

本地一套 `docker-compose.yml` 拉起全部依赖：

```yaml
services:
  frontend:        # Next.js  端口 3000
  backend:         # FastAPI  端口 8000
  worker:          # Celery worker
  postgres:        # PostgreSQL  端口 5432
  redis:           # Redis（Celery broker + 缓存）  端口 6379
  qdrant:          # 向量数据库  端口 6333
  minio:           # 对象存储  端口 9000
  nginx:           # 反向代理  端口 80
```

### 7.2 本地启动步骤

```bash
# 1. 克隆/初始化项目，复制环境变量
cp .env.example .env
#    填入 LLM API Key、数据库密码等

# 2. 构建并启动全部服务
docker compose up -d --build

# 3. 初始化数据库（首次）
docker compose exec backend python -m app.init_db

# 4. 导入法规知识库（首次）
docker compose exec backend python -m app.rag.ingest --dir ./data/regulations

# 5. 查看日志确认启动正常
docker compose logs -f backend

# 6. 访问
#    前端: http://localhost
#    API 文档: http://localhost:8000/docs
```

### 7.3 本地测试要点

- 用真实样本文档（一份合同、一套招采资料）走完整流程，验证解析→校验→报告闭环
- 测试刚性规则的准确率（不依赖 LLM，应稳定）
- 测试 RAG 检索质量：查询法规能否召回正确条款
- 测试联动校验：故意制造招标价与合同价不一致，看能否检出
- 用 `docker compose down` 停止，`docker compose down -v` 清空数据重测

### 7.4 本地与生产环境差异

| 项 | 本地测试 | 生产服务器 |
|----|---------|-----------|
| 对象存储 | MinIO 容器 | MinIO 或云 OSS |
| 域名/SSL | localhost，无 SSL | 配置域名 + SSL 证书 |
| 资源限制 | 不限 | 按 4.2 配置资源上限 |
| LLM | 可用境外 API 测试 | 按数据合规要求决定境内/境外 |
| 数据 | 测试样本 | 客户真实数据 |

保持 `docker-compose.yml` 在本地和服务器尽量一致，差异通过 `.env` 区分。

## 八、服务器部署与交付

测试完成后部署到服务器交付客户。

### 8.1 服务器配置要求（客户提供）

| 配置项 | 最低 | 推荐 |
|--------|------|------|
| CPU | 8 核 | 16 核 |
| 内存 | 16GB | 32GB（向量库 + LLM 推理占内存） |
| 系统盘 | 100GB SSD | 200GB SSD |
| 数据盘 | 200GB | 500GB（文档 + 向量数据） |
| 操作系统 | Ubuntu 22.04 LTS | Ubuntu 22.04 LTS |
| 其他 | Docker + Docker Compose、公网 IP | 域名 + SSL 证书 |

> 若 Embedding/OCR 模型本地部署且数据量大，建议配 GPU（如 NVIDIA T4），否则用 CPU 推理较慢。

### 8.2 部署步骤

```bash
# 1. 服务器安装 Docker 环境
curl -fsSL https://get.docker.com | sh

# 2. 上传项目代码（git clone 或 scp）
git clone <repo> /opt/compliance-agent
cd /opt/compliance-agent

# 3. 配置生产环境变量
cp .env.example .env
#    填入生产数据库密码、LLM Key、域名等

# 4. 配置 Nginx 域名与 SSL（nginx/nginx.conf）
#    放置证书到 nginx/ssl/

# 5. 构建并启动
docker compose up -d --build

# 6. 初始化数据库 + 导入知识库
docker compose exec backend python -m app.init_db
docker compose exec backend python -m app.rag.ingest --dir ./data/regulations

# 7. 验证
docker compose ps          # 全部 healthy
curl https://域名/api/health
```

## 九、数据安全

审计数据高度敏感，遵循以下原则：

- **全部本地化部署**，数据不出境，存储在客户自己的服务器
- **LLM 调用脱敏**：如用境外 API（Claude/GPT），只传待校验的文本片段，不传完整敏感档案；或对敏感数据部分评估国产模型（DeepSeek、通义千问）
- **账号密码由客户自行管理**，开发方不接触、不留存
- **权限分级**：按岗位限制可见的文件分类
- **全程留痕**：操作日志可溯源

> 数据合规边界需在项目启动前与客户书面确认。
