# 内控评价合规审查智能体平台

## 完整技术开发文档 v3.0

> **平台定位：** 站在上级复核方视角，AI 拿着问题清单和法规，对被检查单位的内控评价报告与佐证材料逐项核查，出具核查报告。
>
> **核心价值：** 把原来需要人工数天完成的材料核查，压缩到分钟级。
>
> 开发路线：自研（不依赖 Dify）· 本地 Docker 测试 → 服务器交付

---

## 一、产品逻辑梳理

### 1.1 四步工作流本质

```
第一步：法规库          → AI 的判断依据（上位法、评价办法、编报指南）
第二步：问题清单库      → AI 的检查模板（真实性清单 + 常见问题清单）
第三步：检查报告+佐证材料 → AI 的被检查对象（被审查单位提交的材料）
第四步：输出核查清单    → AI 的成果（一单位一报告）
```

### 1.2 AI 扮演的角色

**不是**填报辅助工具（站在填报单位角度帮它打分）

**而是**上级复核审查官（站在主管部门/财政部门角度，审查填报单位材料是否真实、合规、完整）

### 1.3 检查维度（来自客户清单）

**维度一：总体合规性**
- 材料真实性（非 PS、非截图、非篡改、可追溯原件）
- 材料年度一致性（必须为 2025 年度，无跨年替代）
- 材料正式性（非草稿、非白条、非非正式纪要）
- 材料要素完整性（公章、签字、页码、关键页齐全）

**维度二：相关性核查（按 6 大业务分类）**
- 组织层面：三重一大、轮岗、内控会议、内控培训
- 预算业务：预算制度、公开、绩效管理
- 收支业务：票据台账、支出审批
- 政府采购：采购全流程文件
- 资产/建设/合同：盘点表、项目全流程、合同台账
- 内部监督：检查方案、整改闭环

**维度三：评分合规性（常见错误审查）**
- 关键岗位轮岗扣分是否合规
- 不适用业务处理是否正确
- 整改状态填报是否按时点判断
- 补充指标是否符合规范（总分必须 10 分）

**维度四：复核规范性**
- 复核权限是否越权/缺位
- 复核岗位是否分离
- 复核内容是否完整
- 复核流程是否合规

**维度五：报告编报合规性**
- 报告基本信息是否准确
- 内控情况描述是否真实
- 问题整改填报是否如实
- 系统操作是否规范（账号、树形结构、签章）

## 三、核心模块详细设计

### 3.1 模块一：法规知识库

**内容组成：**

| 知识库 | 具体内容 | 分块策略 |
|--------|---------|---------|
| 上位法库 | 《行政事业单位内部控制规范》《政府采购法》《政府会计制度》等 | 按条款切分，metadata 带条款号 |
| 评价办法库 | 《行政事业单位内部控制评价办法》 | 按条款切分 |
| 编报指南库 | 财政部 2025 年编报指南（附件1单位指标+附件2部门指标） | 按指标切分，metadata 带指标ID、扣分细则、常见扣分情形 |
| 高频问题库 | 客户提供的常见问题清单（全文结构化存储） | 按问题类型切分 |

**关键设计：附件1/2的常见扣分情形是 RAG 的黄金数据**
每个指标的"常见扣分情形举例"要单独存储为独立 chunk，检索时精确召回，作为 AI 判断的参照标准。

**更新机制：** 支持管理员随时上传新文件，自动触发解析→分块→向量化入库，无需重启服务。

---

### 3.2 模块二：问题清单库

**内容来源：**
- 客户提供的佐证材料核查清单（真实性+相关性清单，按6大业务分类）
- 客户提供的常见问题清单（评分问题+复核问题+报告问题）
- 可持续补充历年审计发现、复核意见

**存储结构：**

```json
{
  "check_item_id": "ZS-001",
  "category": "总体合规性",
  "subcategory": "真实性",
  "check_description": "材料是否加盖公章、签字齐全、要素完整",
  "applicable_indicators": ["1-1-1", "1-2-3"],
  "risk_level": "high",
  "common_patterns": [
    "会议纪要缺少参会人员签字",
    "制度文件无发文日期",
    "审批单签字不完整"
  ],
  "check_method": "rule",  // rule(规则检查) or llm(AI判断)
  "keywords": ["公章", "签字", "盖章"]
}
```

**两类检查方法：**
- `rule`：确定性规则判断（如检测关键词是否存在、日期格式是否正确），不走 LLM，快且准
- `llm`：需要理解语义的判断（如"会议内容是否与内控相关"），走 LLM

### 3.3 模块三：材料接收与解析

**输入范围（客户已明确）：电子文件为主**

| 文件类型 | 解析工具 | 特殊处理 |
|---------|---------|---------|
| PDF（电子版） | PyMuPDF | 提取文本 + 表格结构 + 页码信息 |
| Word（.docx） | python-docx | 提取正文 + 表格 + 标题层级 |
| Excel（.xlsx） | openpyxl | 提取所有 Sheet 的结构化数据 |
| 图片 PDF（扫描件） | PaddleOCR | OCR 后按页存储，标注"扫描件" |

**解析输出统一结构（位置信息贯穿全流程）：**

```json
{
  "file_id": "f-001",
  "file_name": "三重一大制度.docx",
  "file_type": "docx",
  "is_scanned": false,
  "pages": [
    {
      "page": 1,
      "section": "第一章 总则",
      "content": "...",
      "has_signature": true,
      "has_seal": true
    }
  ],
  "tables": [],
  "key_elements": {
    "has_official_seal": true,
    "has_signature": true,
    "issue_date": "2025-03-15",
    "document_number": "XXX发[2025]5号"
  }
}
```

`key_elements` 由规则引擎自动提取，用于真实性核查的刚性判断。

**材料与指标的绑定：**

用户上传材料时需指定该材料对应哪个评价指标，系统记录绑定关系。这是相关性核查的基础。

### 3.4 模块四：AI 核查引擎（核心）

这是整个系统最重要的模块，分为两条处理路径：

#### 路径A：刚性规则检查（不走 LLM，快）

针对能用规则判断的问题清单条目：

```python
# 示例：要素完整性检查
def check_required_elements(file_parsed, indicator):
    issues = []
    
    # 检查公章
    if not file_parsed["key_elements"]["has_official_seal"]:
        issues.append({
            "type": "missing_seal",
            "description": "材料未加盖公章",
            "risk_level": "high",
            "location": "全文"
        })
    
    # 检查发文日期
    if not file_parsed["key_elements"]["issue_date"]:
        issues.append({
            "type": "missing_date",
            "description": "材料无发文日期，无法确认是否为2025年度材料",
            "risk_level": "high"
        })
    
    # 检查年度一致性
    if file_parsed["key_elements"]["issue_date"]:
        year = extract_year(file_parsed["key_elements"]["issue_date"])
        if year != 2025:
            issues.append({
                "type": "wrong_year",
                "description": f"材料日期为{year}年，非2025年度材料",
                "risk_level": "high"
            })
    
    return issues
```

刚性规则覆盖的检查项（不走 LLM）：
- 公章/签字检测
- 年度一致性（日期提取）
- 文件编号格式
- 必要材料缺失（某指标规定必须有会议纪要，但用户上传的是制度文件）
- 补充指标总分校验（是否等于 10 分）
- 整改状态时点判断

#### 路径B：LLM 语义检查（走 AI，深度分析）

针对需要理解内容的检查项：

**Prompt 设计（结构化输出为 JSON）：**

```
你是一名专业的行政事业单位内控评价审查专家。
请对以下材料进行核查，判断其是否满足对应评价指标的要求。

【评价指标】
{indicator_content}

【评分细则】
{deduct_rules}

【常见扣分情形】
{common_deductions}  ← 来自RAG检索附件1/2

【相关法规条款】
{legal_basis}  ← 来自RAG检索法规库

【被核查材料内容】
{material_content}

【核查问题清单】
{check_items}  ← 来自问题清单库

请严格按照以下JSON格式输出核查结果，不得添加任何额外内容：

{
  "overall_judgment": "pass|warning|fail",
  "findings": [
    {
      "check_item_id": "对应的问题清单ID",
      "finding_type": "真实性问题|相关性问题|完整性问题|合规性问题",
      "severity": "high|medium|low",
      "description": "具体问题描述（客观、简洁、可引用）",
      "evidence_location": "问题在材料中的具体位置（页码/章节）",
      "legal_basis": "对应法规条款（只能引用检索到的内容，不得编造）",
      "suggestion": "整改建议"
    }
  ],
  "missing_materials": ["缺少的材料清单"],
  "irrelevant_materials": ["与指标不相关的材料清单"]
}

注意：
1. 只引用上方提供的法规内容，不得编造任何法规条款
2. 问题描述必须引用材料中的具体内容或位置
3. 如材料满足要求，findings 为空数组
```

#### 核查任务调度（异步执行）

一个单位的材料可能有几十个文件、几十个指标，全部串行处理会很慢。

**并行策略：**
- 同一单位的不同指标核查任务并行执行（Celery 并发）
- 刚性规则检查先跑（毫秒级），LLM 检查后跑
- 前端实时显示进度（WebSocket 推送）

```
任务创建
   ↓
拆分为 N 个子任务（每个指标一个子任务）
   ↓
Celery 并发执行
   ├── 子任务1：指标1 刚性检查 + LLM检查
   ├── 子任务2：指标2 刚性检查 + LLM检查
   ├── ...
   └── 子任务N：指标N 刚性检查 + LLM检查
   ↓
所有子任务完成后聚合结果
   ↓
生成核查报告
```

### 3.6 模块六：核查报告生成

**一单位一报告，格式：**

```
封面
  被检查单位名称
  检查日期
  检查依据（法规清单）

第一部分：总体核查结论
  核查结论（通过/有条件通过/不通过）
  问题分布统计表
  核查说明

第二部分：佐证材料总体合规性核查
  2.1 真实性核查结果
  2.2 年度一致性核查结果
  2.3 正式性核查结果
  2.4 要素完整性核查结果

第三部分：各指标相关性核查明细
  按组织层面/业务层面/内部监督分类
  每个问题：问题描述 | 材料位置 | 法规依据 | 风险等级 | 整改建议

第四部分：高频风险点汇总
  真实性高风险（PS/篡改/缺章缺字）
  相关性高风险（材料与指标不匹配）
  完整性高风险（关键材料缺失）

第五部分：整改建议清单
  按优先级排序（高风险优先整改）
  每条建议含整改要点和建议完成时限

签章页
```

**导出格式：** Word（.docx），前端 docx-js 生成，无需后端参与。

### 3.7 模块七：协同复核流程

```
AI 初核完成（自动）
       ↓
人工审查员复核（标注确认/忽略/调整AI发现）
       ↓
生成正式核查报告（含人工复核意见）
       ↓
报告下发被检查单位
       ↓
被检查单位填写整改说明
       ↓
审查员确认整改闭环
```

**权限设计：**

| 角色 | 权限 |
|------|------|
| 超级管理员 | 全部权限，管理知识库、用户、任务 |
| 审查员 | 创建任务、查看结果、复核标注、生成报告 |
| 被检查单位 | 上传材料、查看本单位核查结果、填写整改说明 |
| 只读用户 | 查看报告，不可操作 |

## 四、开发阶段计划

### 阶段一：基础闭环（约 5 天）

**目标：** 跑通单个指标的完整核查链路

1. 法规知识库搭建（导入编报指南附件1/2，这是最重要的数据）
2. 问题清单库结构化（把客户发来的清单录入数据库）
3. 文档解析模块（PDF + Word + Excel 解析，输出统一结构）
4. 刚性规则检查引擎（公章检测、年度检查、关键词检测）
5. 单个指标的 LLM 核查（Prompt 调试到输出稳定 JSON）
6. 简单的前端上传界面 + 结果展示

**验收标准：** 上传一份"三重一大"制度文件，AI 能输出结构化核查意见。

### 阶段二：全量指标覆盖（约 5 天）

**目标：** 覆盖所有评价指标，并行处理

1. 全量评价指标库录入（附件1单位指标 + 附件2部门指标）
2. 问题清单与指标的映射关系建立
3. Celery 并发任务调度（多指标并行核查）
4. 前端核查进度实时展示（WebSocket）
5. 核查结果聚合与展示页面

**验收标准：** 上传一个完整单位的所有材料（10+ 文件，30+ 指标），5 分钟内完成核查并展示结果。

### 阶段三：报告生成与复核（约 4 天）

1. Word 核查报告生成（docx-js，覆盖全部章节格式）
2. 人工复核标注功能（确认/忽略/调整）
3. 整改跟踪模块
4. 报告下发和整改回填流程

**验收标准：** 生成的 Word 报告格式规范，包含全部章节，问题定位可溯源。

### 阶段四：优化与交付（约 3 天）

1. 知识库管理界面（管理员可自行更新法规/问题清单）
2. 历史任务查询
3. 批量处理（同时核查多个单位）
4. 性能优化
5. 本地测试完整走通 → 部署服务器

**总工期：约 17 个工作日**

## 五、关键技术难点与对策

| 难点 | 对策 |
|------|------|
| **材料与指标绑定关系** | 上传时要求用户指定材料对应指标，不能全部丢进去让 AI 猜 |
| **扫描件 OCR 精度** | 客户已说明主要是电子文件，OCR 是兜底，优先处理电子版 |
| **AI 编造法规条款** | Prompt 强约束 + RAG 检索结果作为唯一引用来源，输出中法规引用必须带原文片段 |
| **长文档超上下文** | 材料按页/章节分块，只把与当前指标相关的段落送入 LLM，控制单次 token 量 |
| **核查结果一致性** | 对同一材料多次核查结果可能略有差异，人工复核标注是最终确认步骤 |
| **多文件关联核查** | 某些问题需要跨文件比对（如制度文件和执行记录是否配套），设计跨文件关联检查任务 |
| **真实性风险（PS/篡改）** | LLM 无法检测图像层面的 PS，需明确告知：AI 核查的是文本内容合规性，原件真实性需人工验原件 |

---

## 六、一个需要明确的边界

关于**材料真实性（PS/篡改/原件可追溯）**这一块，需要和客户说清楚：

- AI **能做的**：检查文本内容的逻辑一致性（日期前后矛盾、数字逻辑错误、内容套模板等）
- AI **做不到的**：识别图片层面的 PS 痕迹、验证是否为原件

建议在核查报告中注明："本报告对材料内容进行逻辑核查，原件真实性需结合现场核查或档案比对确认。"

## 附录：DeepSeek V4 Pro 接入说明

### 模型基本信息

DeepSeek V4 Pro 是 1.6T 参数（49B 激活）的 MoE 大模型，支持 1M token 超长上下文窗口，专为复杂推理、长文档分析和 Agent 工作流设计。

API 模型字符串：`deepseek-v4-pro`，自 2026 年 5 月 31 日起价格永久调整为原价的 1/4，是目前全球定价最低的顶级大模型之一。

**对本项目的优势：**
- 1M 上下文窗口：一次性处理整份内控报告+多份佐证材料，无需频繁分块
- 国内访问稳定：相比境外 API 不受网络波动影响，数据也更易满足合规要求
- 价格极低：大量调用的审计场景成本可控
- 兼容 OpenAI 格式：接入成本极低，代码几乎不用改

### 三种推理模式选择

DeepSeek V4 Pro 支持三种推理模式：Non-think（快速响应）、Think High（逻辑分析）、Think Max（最大推理努力）。

按本项目不同场景选择：

| 使用场景 | 推荐模式 | 理由 |
|---------|---------|------|
| 刚性规则检查（公章/日期/关键词） | 不走 LLM | 规则引擎直接处理，不消耗 token |
| 材料相关性核查（内容是否匹配指标） | Non-think | 速度快，内容理解不需要深度推理 |
| 复杂合规判断（如"制度条款是否覆盖所有环节"） | Think High | 需要逻辑推导，但不必到最高档 |
| 跨文件联动核查（制度+执行记录配套分析） | Think High / Think Max | 需要关联推理 |

### API 调用代码示例

```python
from openai import OpenAI  # DeepSeek 兼容 OpenAI 格式

client = OpenAI(
    api_key="your-deepseek-api-key",
    base_url="https://api.deepseek.com/v1"
)

# Non-think 模式（快速，适合相关性核查）
response = client.chat.completions.create(
    model="deepseek-v4-pro",
    messages=[
        {"role": "system", "content": "你是内控评价合规审查专家..."},
        {"role": "user", "content": prompt}
    ],
    response_format={"type": "json_object"},  # 强制 JSON 输出
    temperature=0.1  # 审查任务要求确定性高，温度调低
)

# Think High 模式（复杂判断）
response = client.chat.completions.create(
    model="deepseek-v4-pro",
    messages=[...],
    extra_body={"thinking": {"type": "enabled", "budget_tokens": 8000}},
    response_format={"type": "json_object"},
    temperature=0.1
)

result = response.choices[0].message.content
```

### LLM 适配层设计（支持后续切换）

在 `backend/app/llm/client.py` 封装为统一接口，模型通过环境变量切换：

```python
# .env 配置
LLM_PROVIDER=deepseek
LLM_MODEL=deepseek-v4-pro
LLM_API_KEY=your-api-key
LLM_BASE_URL=https://api.deepseek.com/v1

# 如需切换国产备选（防止 DeepSeek 服务波动）
# LLM_PROVIDER=qwen
# LLM_MODEL=qwen-max
```

这样即使后续需要换模型，只改环境变量，业务代码不动。
